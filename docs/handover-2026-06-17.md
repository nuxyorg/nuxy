# Handover — Kalan Büyük Roadmap Maddeleri (2026-06-17)

Bu doküman `docs/roadmap.md`'deki kalan büyük işleri **paralel** çalışılabilir parçalara böler.
Her track bağımsız dosya setine dokunur — aynı anda birden fazla agent/kişi çakışmadan çalışabilir.

> Önce oku: `docs/roadmap.md` (genel takip listesi), `CLAUDE.md` (TDD zorunlu, komutlar), `rules/AGENTS.md`.
> Bugün (2026-06-17) roadmap'teki şu maddeler zaten bitti, tekrar açma: Button/Modal rewrite, omnibar/command-palette
> a11y+CSS, shared utils (`parse.ts`, `mirror-attrs.ts`, `focus-trap.ts`), worker `markFailed`, RadioGroup (native
> davranış zaten yeterli), tema JSON semantic tokenları.

## Genel kurallar

- TDD zorunlu (CLAUDE.md) — yeni davranış eklerken önce test yaz.
- Her track'in sonunda **kendi dokunduğu dosyalarla sınırlı** `pnpm -C src test` + `pnpm typecheck` + `pnpm lint`
  çalıştır. Track'ler bitince tam `pnpm check` ile son kontrol yap.
- Görsel/davranışsal regresyon riski olan değişikliklerde (CSS, Modal/SelectBox gibi UI) mutlaka önce/sonra
  karşılaştırması yap (gerekirse `/run` skill ile uygulamayı açıp gözle kontrol et).
- Roadmap'i güncel tut: bitirdiğin maddeyi `docs/roadmap.md`'de `[x]` yap, kısa bir not ekle (ne yapıldı, hangi dosya).

## Track A — `scanner.ts` 5 modüle bölünmesi

**Dosya:** `src/electron/extensions/scanner.ts` (652 satır, tek dosya)
**Dokunulacak alan:** yalnız `src/electron/extensions/` altında yeni dosyalar + `scanner.ts`'den taşıma.
**Diğer track'lerle çakışma riski:** yok (kimse bu dosyaya dokunmuyor).

Mevcut fonksiyon haritası (satır numaraları güncel):

```
scanner.ts
  detectNodeImports()              L67   → manifest-loader.ts adayı (Node import tespiti, güvenlik taraması)
  scanDirectoryForNodeImports()    L111  → manifest-loader.ts
  rescanExtensions()               L161  → index.ts (orchestration)
  startExtensionWatcher()          L173  → dev-sync.ts
  restoreWritable()                L179  → worker-manager.ts ya da security yardımcı
  dedupeExtractedByManifestId()    L190  → manifest-loader.ts
  verifyAndSecureExtension()       L236  → manifest-loader.ts (imza/izin doğrulama)
  registerExtensionByType()        L306  → worker-manager.ts (tipe göre worker/tema/icon kaydı dispatch)
  promptTrustPublisherKey()        L383  → manifest-loader.ts
  scanExtensions()                 L404  → index.ts (orchestration, ana giriş noktası)
```

Hedef yapı (roadmap'te zaten tanımlı):

```
src/electron/extensions/
  manifest-loader.ts    # JSON parse + Zod validation + imza doğrulama
  worker-manager.ts     # spawn, terminate, timeout, registerExtensionByType
  theme-registrar.ts    # zaten themes/extension-themes.ts var — burada sadece scanner'ın çağrı noktasını ayır
  icon-registrar.ts     # zaten icons/registry.ts var — aynı şekilde
  dev-sync.ts           # startExtensionWatcher + dev mode kopyalama
  index.ts              # sadece orchestrate eder (scanExtensions, rescanExtensions)
```

**Not:** `theme-registrar.ts`/`icon-registrar.ts` için kayıt mantığının kendisi zaten ayrı dosyalarda
(`src/electron/themes/extension-themes.ts`, `src/electron/icons/registry.ts`). Burada yapılacak iş, scanner.ts'in
bunlara yaptığı çağrıları temiz bir arayüz arkasına almak, scanner.ts'i küçültmek — yeniden yazmak değil.

**Risk:** Bu dosya extension yükleme bootstrap'inin kritik yolu. `scanExtensions()` test kapsamı zayıfsa (kontrol et:
`src/electron/extensions/*.test.ts`), bölmeden önce karakterizasyon testleri ekle. Mevcut testler:
`extension-reload.test.ts`, `registry.test.ts`, `seed-bundled.test.ts` — `scanner.ts`'in kendisi için ayrı bir
`scanner.test.ts` yok, bu da ilk adım olmalı.

**Kabul kriteri:** `scanner.ts` sadece orchestration + re-export içerir (<100 satır), her yeni modülün kendi testi var,
`pnpm -C src test` ve full app boot (manuel ya da e2e) hâlâ çalışıyor.

## Track C — CSS token fallback temizliği

**Dosyalar (11 component × 2 dosya = 22 dosya, ~60 occurrence):**

```
ChatMessage/{index.css,nuxy-chat-message.ts}
ConversionCard/{index.css,nuxy-conversion-card.ts}
ItemLeading/{index.css,nuxy-item-leading.ts}
MarkdownEditor/nuxy-markdown-editor.ts
MarkdownText/{index.css,nuxy-markdown-text.ts}
MediaPreview/{index.css,nuxy-media-preview.ts}
PropertiesPanel/{index.css,nuxy-properties-panel.ts}
SectionHeader/{index.css,nuxy-section-header.ts}
ShortcutHint/{index.css,nuxy-shortcut-hint.ts}
TabBar/{index.css,nuxy-tab-bar.ts}
TwoPanel/{index.css,nuxy-two-panel.ts}
WizardSection/{index.css,nuxy-wizard-section.ts}
```

(Hepsi `extensions/ui-default/src/components/` altında.)

**Dokunulacak alan:** yalnızca yukarıdaki 22 dosya. Aralarında ortak bağımlılık yok, **bu liste kendi içinde de 2-3
kişiye bölünebilir** (örn. ChatMessage+ConversionCard+ItemLeading+MarkdownEditor / MarkdownText+MediaPreview+
PropertiesPanel / SectionHeader+ShortcutHint+TabBar+TwoPanel+WizardSection).

**İş:** `var(--token, rgba(...))` kalıbındaki fallback değerlerini kaldır — token zaten
`extensions/ui-default/src/styles/base.css`'te tanımlı (örn. `--text-primary`, `--text-muted`, `--border`,
`--hover`, `--accent-subtle`, `--border-subtle`). Önce her token'ın base.css'teki canonical değeriyle inline
fallback'in **aynı** olduğunu doğrula (`grep -n "token-adı:" extensions/ui-default/src/styles/base.css`); eşitse
fallback'i güvenle silebilirsin (`var(--text-primary, rgba(255,255,255,0.92))` → `var(--text-primary)`). Eşit
değilse şaşırtıcı bir uyumsuzluk var — silmeden önce sebebini araştır, görsel regresyon riski var.

**Ek (opsiyonel, düşük öncelik):** `SelectBox/index.css` ve diğer component'lerdeki saf `rgba(0,0,0,0.X)`
box-shadow değerleri (token değil, dekoratif gölge) — `nuxy-shell.css`'te yapılan `var(--shadow-dark, rgba(...))`
deseniyle eşleştirilebilir ama bu roadmap'in asıl istediği şey değil, atlanabilir.

**Kabul kriteri:** `grep -rE "var\(--[a-zA-Z0-9-]+,\s*rgba?\(" extensions/ui-default/src` boş döner (ya da sadece
kasıtlı olarak token'sız bırakılanlar — onları da bir yorumla işaretle). Görsel fark yok (manuel kontrol).

## Track D — Kernel IPC handler test kapsamı

**Dosyalar (test eksik olan 4 modül, her biri bağımsız):**

```
src/electron/ipc/kernel-handlers/extensions.ts   → extensions.test.ts (yok)
src/electron/ipc/kernel-handlers/i18n.ts         → i18n.test.ts (yok)
src/electron/ipc/kernel-handlers/system.ts       → system.test.ts (yok)
src/electron/ipc/kernel-handlers/themes.ts       → themes.test.ts (yok)
```

`composition.ts` için zaten `src/electron/ipc/composition.test.ts` var — referans pattern olarak kullan.
`register.test.ts` ve `validate.test.ts` genel routing/validation'ı test ediyor, handler-spesifik mantığı değil.

**Dokunulacak alan:** her kişi kendi handler dosyasının yanına kendi test dosyasını ekler — sıfır çakışma.

**İş:** Roadmap'in istediği matris her handler fonksiyonu için: happy path, missing params, malformed payload,
extension not found (uygunsa). `extensions.ts`'teki handler'lar için örnek: `listTools`, `setExtensionEnabled`,
`installExtension`, `uninstallExtension` — her biri payload validasyonu + başarı + hata path'i. Mock pattern için
`registry.test.ts` ve `composition.test.ts`'e bak (`vi.mock` ile `loadedExtensions`, `registry.js` mock'lanıyor).

**Kabul kriteri:** 4 yeni test dosyası, her handler fonksiyonu için en az happy-path + 1 hata path testi, `pnpm -C
src test` yeşil.

## Track E — `pressOmnibarKey` gerçek input ile çalışsın

**Dosya:** `extensions/shell/tests/e2e.spec.ts` (helper L30-40)
**Dokunulacak alan:** yalnız bu dosya (ve gerekirse `src/e2e/fixtures.ts` içindeki yardımcılar — ama oraya
dokunmadan önce başka bir track onu kullanıyor mu kontrol et).

**Güncel durum:** Helper `page.evaluate()` içinde `dispatchEvent(new KeyboardEvent(...))` ile sentetik event
gönderiyor — gerçek klavye input'unu simüle etmiyor (focus/IME/composition davranışlarını test etmiyor).

**İş:** Playwright'ın gerçek klavye API'sini kullan: `await page.keyboard.press(key)` (input elementine önce
`page.locator(...).focus()` ya da click ile focus verip). Mevcut tüm çağrı yerlerini (`pressOmnibarKey(appPage,
'ArrowDown')` vb.) bu yeni implementasyonla uyumlu tut — imza aynı kalmalı, böylece çağıran testler değişmesin.

**Kabul kriteri:** Helper artık `page.keyboard.press`/`page.keyboard.type` kullanıyor, mevcut tüm e2e testler
(`pnpm test:e2e:core` ya da ilgili extension e2e komutu) hâlâ geçiyor.

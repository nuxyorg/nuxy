## Issues

### 1. `transition: all` in multiple CSS files

**Problem**: `packages/ui/src/components/Button/index.css:7`, `Card/index.css:7`, `ListItem/index.css:7`, and `extensions/shell/shell.css:139` all use `transition: all 200ms` / `transition: all 300ms`. `transition: all` animates every CSS property including layout-affecting ones.

**Impact**: Forces browser to recalculate layout on every transition tick, causing unnecessary repaints.

**Proposed Fix**: Replace with explicit property list — e.g. `transition: background-color 200ms, border-color 200ms, color 200ms, transform 200ms`.

**Risk Level**: Low

**Applied Automatically?**: No

### 2. Hardcoded `10px` in ShortcutBar

**Problem**: `packages/ui/src/components/ShortcutBar/index.css:6` uses hardcoded `10px` instead of a CSS design token.

**Impact**: Inconsistency — surrounding components use `var(--space-3)` (8px) or `var(--space-4)` (12px). Makes theme changes require hunting raw pixel values.

**Proposed Fix**: Replace `10px` with `var(--space-3)` (closest token).

**Risk Level**: Low

**Applied Automatically?**: No

### 3. Decorative emoji without `aria-hidden`

**Problem**: `extensions/shell/frontend.js:373` — a 🔍 emoji is rendered as decorative UI chrome but has no `aria-hidden="true"`, so screen readers announce it as "magnifying glass tilted left".

**Impact**: Accessibility regression — pollutes screen reader output.

**Proposed Fix**: Add `aria-hidden="true"` to the wrapping span.

**Risk Level**: Low

**Applied Automatically?**: No

### 4. Results list missing ARIA roles

**Problem**: `extensions/shell/frontend.js:400–411` — the results container is a plain `<div>` with no `role="listbox"`. Individual result items have no `role="option"` or `aria-selected` attribute.

**Impact**: Screen readers cannot identify this as a selectable list. Keyboard navigation is invisible to assistive technology.

**Proposed Fix**: Add `role="listbox"` to the container, `role="option"` + `aria-selected={isSelected}` to each item.

**Risk Level**: Medium (requires testing with screen reader)

**Applied Automatically?**: No

### 5. `ListItem` renders clickable `<div>` without keyboard semantics

**Problem**: `packages/ui/src/components/ListItem/index.tsx` — renders a `<div>` with `cursor: pointer` and an `onClick` handler but no `role="button"`, no `tabIndex`, and no `onKeyDown` handler.

**Impact**: Keyboard-only users cannot activate list items. Violates WCAG 2.1 SC 2.1.1.

**Proposed Fix**: Add `role="button"`, `tabIndex={0}`, and `onKeyDown` (Enter/Space trigger onClick).

**Risk Level**: Medium

**Applied Automatically?**: No

### 6. No `prefers-reduced-motion` support

**Problem**: `extensions/shell/shell.css` defines cursor-blink and pulse animations with no `@media (prefers-reduced-motion: reduce)` override.

**Impact**: Users with vestibular disorders or motion sensitivity get animations they've opted out of at the OS level.

**Proposed Fix**: Add:
```css
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    animation-duration: 0.01ms !important;
    transition-duration: 0.01ms !important;
  }
}
```

**Risk Level**: Low

**Applied Automatically?**: No

### 7. Theme variable alignment gap

**Problem**: The theme files (`src/themes/default-dark.json`, `default-light.json`) define `colors` and `tokens`, but the UI components reference CSS custom properties like `--space-3`, `--radius`, `--surface-2` etc. The bridge between JSON theme values and CSS custom properties happens at runtime in `App.tsx` via `Object.entries(colors/tokens)` → `setProperty`. If a theme omits a token a component expects, the component silently falls back to browser defaults.

**Impact**: Silent visual regressions when themes are incomplete.

**Proposed Fix**: Create a type contract (`ThemeTokens` interface in `@nuxy/core`) listing all required CSS variables. Validate theme files against this type at load time and warn on missing tokens.

**Risk Level**: Medium

**Applied Automatically?**: No

### 8. `EmptyState` `error` prop renders raw error string without XSS protection

**Problem**: `packages/ui/src/components/EmptyState/index.tsx` — the `error` prop is rendered as text content, which is safe. However `App.tsx:73` passes `err.message` directly. If an extension ever constructs a malicious protocol URL, the browser could expose a crafted error string.

**Impact**: Low risk in Electron context (renderer is trusted), but worth noting.

**Proposed Fix**: Ensure `error` is always rendered as `{error}` (JSX text node), never as `dangerouslySetInnerHTML`. Verify this is already the case.

**Risk Level**: Low

**Applied Automatically?**: No

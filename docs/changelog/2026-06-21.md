# Changelog — 2026-06-21

## `com.nuxy.store` — ported the legacy extension marketplace to the current architecture

- Full rewrite of the old React-based `store` extension (`hooks/useStoreData.ts`, `useStoreActions.ts`,
  `useStoreDerivedData.ts`, `useStoreKeyboard.ts`, `useStoreSync.tsx`, `components/StoreExtensionList.tsx`,
  `components/StoreExtensionDetail.tsx`) into `extensions/store/` using the current LitElement +
  controller architecture (light DOM, `@nuxyorg/core` `html`, `BaseExtensionController`,
  `ShellAction[]` keyboard wiring via `core.shell.registerShellActions`).
- New files: `manifest.json`, `backend.ts`, `controller.ts`, `frontend.ts`, `types.ts`, `settings.json`,
  `locales/{en,tr,ja}.json`, `utils/storeFilter.ts`, `tests/{backend,controller,storeFilter,nuxy-tool-store}.test.ts`.
- Behavior preserved from the legacy extension: remote registry JSON fetch (configurable `registryUrl`
  setting, default `https://raw.githubusercontent.com/atagulalan/nuxy-assets/main/registry.json`),
  merge with locally installed extensions, tab/search filtering (`all` / `tool` / `theme` / `iconpack` /
  `installed` / `updates`), install/update/uninstall actions, system-extension detection
  (`com.nuxy.shell`, `com.nuxy.settings`, and any extension with `manifest.bootstrap: true` are
  flagged `isSystem` and protected from uninstall in the UI).
- Behavior adjusted for the current UI kit: legacy `Badge` variants (`danger`/`warning`/`success`/`default`)
  mapped to the current `nuxy-tag` variant set (`red`/`orange`/`green`/`default`); legacy sidebar tab
  bar (`TabBar` + custom two-panel nav hook) replaced with `<nuxy-tab-bar orientation="vertical">` +
  `<nuxy-two-panel>`, driven entirely by the controller instead of React state.

### Kernel-dependency verification (important — read before assuming this needs follow-up work)

Verified by grepping `src/electron/` before writing `backend.ts`: kernel-side support for
`core.extensions.invoke('kernel', 'listInstalledExtensions' | 'installExtension' | 'uninstallExtension')`
**already exists and is fully wired** today:

- `src/electron/ipc/broker.ts` routes `targetId === 'kernel'` calls from worker extensions to
  `src/electron/ipc/kernel-invokable.ts` (`callKernelChannel`), which is real, no stub.
- `src/electron/ipc/kernel-invokable.ts` implements `listInstalledExtensions` (returns
  `loadedExtensions`), `installExtension` (delegates to `kernelInstallExtension`), and
  `uninstallExtension` (delegates to `kernelUninstallExtension`) — both defined in
  `src/electron/ipc/extension-ops.ts` with real filesystem behavior (downloads `.nuxyext`, writes to
  `EXTENSION_DIR`, triggers a rescan; uninstall removes the extension's folder/zip and rejects
  `com.nuxy.shell` / `com.nuxy.settings` / bootstrap extensions).
- All three channels are covered by existing kernel-side tests (`kernel-invokable.test.ts`,
  `extension-ops.test.ts`, `kernel-handlers/extensions.test.ts`, `broker.test.ts`, `register.test.ts`,
  `validate.test.ts`).

No kernel-side gap exists for this port — `backend.ts`'s `installExtension` / `uninstallExtension` /
`getExtensions` handlers are full proxies to real, already-shipped kernel functionality, not
placeholders.

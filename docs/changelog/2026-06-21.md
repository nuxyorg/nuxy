# Changelog — 2026-06-21

## `angrysearch` — port to current extension architecture

- Ported the legacy `angrysearch` file-search extension (Voidtools Everything-style, SQLite FTS4-backed) from the old React + hooks architecture into a fresh `extensions/angrysearch/` implementation that follows the current rules (no React, controller + LitElement, light DOM, `tests/` layout).
- New files: `manifest.json`, `backend.ts`, `controller.ts`, `nuxy-tool-angrysearch.ts`, `frontend.ts`, `types.ts`, `settings.json`, `locales/{en,tr,ja}.json`, `tests/backend.test.ts`, `tests/controller.test.ts`, `tests/nuxy-tool-angrysearch.test.ts`.
- Preserved behavior: search-as-you-type with a 3-character minimum and 150ms debounce, LIKE vs. REGEXP (via a custom SQLite `REGEXP` function) query modes, configurable search result limit, periodic background re-indexing on a configurable interval, "open file" / "open containing folder" actions, and the ignored-roots safety list that prevents the indexer from descending into `/proc`, `/dev`, `/sys`, `/snap`, `/run`, `/tmp`, `/var/run`, `/var/lock` when scanning from `/`.
- Replaced the four legacy hooks (`useAngrysearchData`, `useAngrysearchActions`, `useAngrysearchKeyboard`, `useAngrysearchSync`) and the `AngrysearchItem.tsx` component with a single `AngrysearchController` class (state, IPC, keyboard actions via `window.core.shell.registerShellActions`) and a light-DOM `LitElement` (`nuxy-tool-angrysearch`) rendering a single-panel results list with `nuxy-list`/`nuxy-list-item`/`nuxy-empty-state`.
- Manifest permissions trimmed to the minimal real set actually used by the backend: `fs`, `db`, `shell` (the legacy manifest also declared `storage`, which the backend never called).
- Verified `DbHandle.function()` (used to register the custom `REGEXP` SQLite function) and FTS4 virtual tables are both supported by the current `core.db` runtime (`packages/extension-host/src/core-proxy.ts` opens a `node:sqlite` `DatabaseSync`, whose native `.function()` and `fts4` support were confirmed directly) — **no changes to `packages/core` or the runtime were needed**, since the type and implementation already existed.
- `pnpm -C src test`, `pnpm typecheck`, and `pnpm lint:fix` all pass with zero errors in the new files (44/44 new tests green; the only failing tests in the full suite are 3 pre-existing, unrelated failures in `download-manager`, `notes`, and `ui-default/Kbd`).
